## 🧠 穩定決策與共用規則（v1.2 拍板｜2026/06/25）
> 來源：[sean_obsidian_okf_research_report_v1_2](https://app.notion.com/p/38a597010d2c80408866dcdb6e54dc16)。以下為會影響未來所有迭代的穩定規則；細節與理由回溯該報告。
### 1. 三層架構（定案）
- **Notion ＝ 行動／專案／決策控制檯**：任務、deadline、報價追蹤、出貨、客戶狀態、資料表、協作。
- **Obsidian ＝ 長期知識本體（SoT）**：concept／case／people／literature／playbook／principle 卡片＋MOC＋wiki。
- **`_okf/`**** ＝ agent-readable 交換層**：由 AI／script 產生，可刪除重建，**不是第二套筆記**。
### 2. 工具主路徑（定案，取代舊頁 A/B/C 開放選項）
- **主路徑**：Obsidian + Markdown + Claude Code + Git + scripts。
- **Obsidian-side agent**：Claude Code 直接讀寫 `D:\Sean_KB\` 的 markdown；Obsidian URI 僅用於開 note／vault／search。
- **Notion-side bridge**：Notion AI（蒜頭）日常；Notion MCP、`ntn` CLI 中期導入；Notion Workers 列第二階段。
- **明確排除**：local LLM／Ollama（**永久排除**）；向量庫／RAG 當主路徑、graph view 當成功指標（本階段排除）。詳細理由與封存項目見 [Backlog Page](https://app.notion.com/p/5deb4cdb4ba5417da722a2002409b875)。
### 3. Vault 結構與位置
- **位置**：`D:\Sean_KB\`（放 D 槽；C 槽留系統與開發環境）。
- **骨架**：`_inbox / sources / notes / maps / wiki / _system / .claude / CLAUDE.md / _okf`（細節見報告 §5.2）。
- **Note types**：Concept／Case／Person／Literature／Playbook／Principle／Map／Source／Decision。
### 4. 卡片與 frontmatter 規則
- 一張卡只講一個核心想法；用自己的話寫；能離開原語境仍成立。
- promoted note 必備 OKF core fields：`type`、`title`、`description`、`timestamp`（＋Sean custom fields：`id`、`domain`、`status`…）。
- raw transcript／Excel／ERP／任務狀態 **不進** Obsidian，只留 `sources/` 或 Notion。
### 5. Notion ↔ Obsidian 橋接
- **雙向異步引用，不做全文同步**：Notion 專案頁放 Knowledge References（`obsidian://` 連結）；Obsidian note frontmatter 放 `notion_refs`。
- 專案完成後才萃取：Notion 完成頁 → 去識別化 → Obsidian draft → Sean review → export `_okf/` → Notion 回寫 `knowledge_extracted = true`。
### 6. 資料分級（sensitivity）— 已廢止（2026/06/26）
> sensitivity 分級系統已移除（見〈決策紀錄〉2026/06/26）。原 public／internal／private／confidential 分級與 export 限制一併廢止，不再生效。
### 7. 維護分工（AI 維護、人類 curate）
- **AI（蒜頭／Claude Code）**：索引、補 link、整理 MOC、產 index、export OKF、跑 content lint／OKF lint；改動一律走 Git diff。
- **Sean**：判斷哪些值得留、promote、approve、最終決策。
### 8. 主題與來源是「導航層」，不是物理結構（2026/06/25 補）
- **分層原則**：原子卡採 Zettelkasten 扁平存放，一卡一想法，**檔名以概念／主張命名**（或穩定 id），不以主題或來源命名。
- **主題與來源不作為資料夾或檔名的長期分類依據**，一律進入 `frontmatter`、`tags`、MOC、wikilinks 與 LLM Wiki 合成頁。
- **主題昇華時新增或重組 MOC（append-only 樹），不搬動底層卡片**（例：`希臘羅馬神話-MOC` → 之後加 `神話學-MOC` 當父節點，卡片不動）。
- **身份錨點是 ****`pkm-id`**** ＋ frontmatter**，不是檔名或資料夾路徑（呼應 §5.5「note 可改名，`id` 不變」）。
- **跨域多重歸屬**靠 tags／多 MOC：一張卡可同時掛在多個主題下，不被資料夾邊界切斷（這正是跨領域思考最值錢的地方）。
- **來源前綴僅允許作為遷移期的暫時防撞措施，最終回歸 metadata。**
### 9. 資料夾只切「型別」，不切「領域」（2026/06/25 補）
- **可切的軸（資料夾／實體結構）**：僅限**型別**這種有限、穩定、封閉的軸——`notes/` 下 `concepts/`、`cases/`、`people/`、`literature/`、`playbooks/`，每個型別資料夾內**扁平**。
- **不可切的軸**：領域、主題、來源（會成長／昇華／跨域）一律進 `domain`／`tags` frontmatter + MOC，**不開資料夾**。
- **收掉領域味資料夾**：`work-insights/`、`parenting-insights/`、`investment-logic/`、`life-principles/` 折回 `concepts/`／`cases/` + `domain:` frontmatter + MOC（例：投資邏輯卡 = `concepts/` 的 Concept note，`domain: investment`，由 `投資-MOC` 連入）。
- **禁止雙層巢狀（領域×型別）**，那是過度設計。
- **`maps/`**** 一律扁平**：裡面只有一種型別（Map／MOC），沒有型別軸可切；唯一能切的軸是領域，而領域正是本條禁止的軸。MOC 階層改用 **MOC 互連**（父 MOC 連子 MOC，如 `神話學-MOC` → `希臘羅馬神話-MOC`）表達，不用巢狀資料夾——跨域 MOC（最高價值的那批）才不會被資料夾邊界鎖死。
- **通用判準**：某一層只在「存在 ≥2 個穩定型別需要分開」時才開資料夾；若該層只有單一型別（如 `maps/` 全是 MOC），一律扁平。
- 一句話：**資料夾只允許切有限且穩定的結構軸（型別）；任何會成長／跨域的軸進 frontmatter + MOC。**
