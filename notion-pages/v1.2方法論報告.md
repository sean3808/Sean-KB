## 1. 執行摘要：Sean-KB 的最終架構判斷
Sean 要建立的不是普通筆記庫，而是一套能長期支撐以下事情的個人 knowledge operating system：
- PMBA 學習、課堂討論、管理理論、個案反思。
- 傳統產業出口物流、ERP、排貨、客戶流程、自動化與數位轉型。
- AI 工具整合、Claude Code、ChatGPT、Notion AI、Python / PowerShell / GAS 腳本。
- 家庭、育兒、睡眠、健康、生活決策。
- 未來從千旺體系轉向「business-process concretization + AI tooling integration」的職涯能力。
最終決策如下：
```plain text
Notion：行動與決策控制檯
Obsidian：長期知識本體
OKF：agent-readable exchange layer
Claude Code：本地 filesystem agent
Notion MCP / ntn / Workers：Notion-side agent bridge
ChatGPT：研究、推理、報告與架構設計
Sean：curator、判斷者、決策者
```
### 1.1 一句話結論
> **Obsidian 是 Sean 的知識工作區；OKF 是 agent 可交換的知識封裝；Notion 是行動與決策控制檯。**
### 1.2 不做什麼，比做什麼更重要
本報告明確不建議：
- 不把 Obsidian 變成 Notion。
- 不把 Notion 變成長期知識庫。
- 不做 Notion ↔︎ Obsidian 內容雙向同步。
- 不把 raw operational data 全部丟進 Obsidian。
- 不導入本地 LLM / Ollama / local RAG 作為主路徑。
- 不以漂亮 graph view 作為成功指標。
- 不讓 Sean 承擔大量手工整理成本。
- 不把 OKF 變成第二套筆記系統。
### 1.3 成功定義
90 天後，Sean-KB 的成功不是「有很多筆記」，而是：
1. PMBA 每堂課後能穩定產出少數可長期使用的 concept / case notes。
2. 工作事件能萃取成流程洞察、組織洞察、決策約束與自動化 playbook。
3. Notion 專案頁能引用 Obsidian / OKF 的知識，不再每次從零思考。
4. Claude Code 可以定期幫忙整理、lint、產生 index、export OKF。
5. 知識庫不因資訊增加而腐爛，反而越用越有結構。

### 2.3 設計原則
1. **Agent-friendly 優先於 human-friendly**
	不是因為人不重要，而是因為 Sean 不想手工維護太多；資料結構要讓 agent 能讀、能改、能檢查。
2. **Filesystem-native 優先**
	Obsidian 的最大優勢不是 UI，而是 Markdown files 直接存在本地 filesystem。
3. **控制檯與知識庫分離**
	Notion 管任務、專案、狀態；Obsidian 管長期知識。
4. **引用，不同步**
	Notion ↔︎ Obsidian 雙向引用即可，不做全文雙向同步。
5. **原始資料與萃取知識分離**
	transcript、Excel、ERP、客戶表、發票、出貨狀態不是知識本體；它們是萃取素材。
6. **AI 維護，人類 curate**
	AI 負責整理、織網、補連結、跑 lint；Sean 負責判斷哪些東西值得留下。
7. **先建立未來規則，不急著清舊資料**
	第 1 個月不搬家、不整理全部歷史，只讓新進知識用正確流程進來。

### 3.1 Zettelkasten 在 Sean-KB 中的角色
Zettelkasten 不是要 Sean 手寫大量卡片，而是提供內容品質原則：
- 一張 concept note 只處理一個核心想法。
- 用自己的話重寫，而不是複製原文。
- 每張 note 要能離開原始語境仍然有意義。
- note 應該能連到其他概念與案例。
在 Sean 的情境中，Zettelkasten 不應變成卡片潔癖，而是防止 Obsidian 變成資料垃圾場。
### 3.2 LYT / MOC 在 Sean-KB 中的角色
LYT / MOC 的價值在於導航，而不是分類表演。
Sean-KB 的 MOC 應該服務這些主題：
- PMBA 管理學習地圖。
- 出口物流與船務流程地圖。
- AI 自動化與 agent tooling 地圖。
- 傳統產業數位轉型地圖。
- 育兒與家庭決策地圖。
- 個人職涯轉型地圖。
MOC 不是資料夾索引，而是「我現在要理解這個領域，該從哪裡進去」。
### 3.3 Karpathy LLM Wiki 對 Sean 的啟發
Karpathy 的 LLM Wiki pattern 把知識庫分成三層：raw sources、LLM-generated wiki、schema / CLAUDE.md。它的核心不是 RAG，而是讓 LLM 逐步建立一個 persistent, compounding artifact：新資料進來時，LLM 不只是索引，而是整理、更新、交叉引用、標註矛盾，並把答案反向寫回 wiki。[Source: Karpathy LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)
對 Sean 來說，這個概念很關鍵：
```plain text
raw sources：PLAUD、Reader、PDF、Notion project、Excel 摘要
wiki：Obsidian promoted notes / maps / wiki pages
schema：CLAUDE.md + _system/schemas + _system/prompts
```
但 Sean-KB 不能完全照抄 Karpathy 的「LLM owns wiki layer」：
- Sean 的 Obsidian 是長期知識 SoT，不應完全由 LLM 亂改。
- AI 可以產生草稿、補 link、整理 MOC、export OKF。
- promoted note 的內容品質仍需 Sean curate。
因此，Sean-KB 應改寫為：
```plain text
LLM drafts and maintains structure.
Sean curates and approves knowledge.
```
### 3.4 OKF 對 LLM Wiki 的標準化價值
Google Cloud 2026-06-13 發布 OKF 時，把 OKF 定位為把 LLM-wiki pattern formalize 成 portable、interoperable format 的開放規格。OKF v0.1 將 knowledge 表示為 directory of markdown files with YAML frontmatter，並使用少量欄位：`type`、`title`、`description`、`resource`、`tags`、`timestamp`。[Source: Google Cloud OKF Blog](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing)
OKF spec 本身強調：它是 minimal format，不是平台；沒有 schema registry、central authority、required tooling；能 `cat` 就能讀，能 `git clone` 就能 ship。[Source: OKF SPEC](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)
這讓 Sean-KB 的結論更清楚：
> **Obsidian 可以保留 native 使用方式；OKF 則作為 generated exchange layer，讓 agent 以標準格式讀懂 Sean 的知識。**

### 4.2 Notion 的權責
Notion 是控制檯，適合有狀態、有 deadline、有資料表、有協作、有 action 的東西。
應留在 Notion：
- PMBA 課程主控台。
- PMBA 作業 deadline。
- 專案看板。
- 任務與週計畫。
- 報價後追蹤總覽。
- 出貨進度。
- 客戶狀態。
- 工作決策紀錄。
- 家庭或生活專案。
- 需要表格篩選、狀態更新、多人協作的內容。
Notion 不應承擔：
- 長期永久筆記。
- 跨年度管理洞察。
- 概念網路。
- 工作案例萃取後的方法論。
- local-first knowledge archive。
### 4.3 Obsidian 的權責
Obsidian 是長期知識本體，適合保存 Sean 已經判斷「未來會反覆使用」的知識。
應進 Obsidian：
- PMBA 課堂萃取出的管理概念。
- 工作事件萃取出的流程洞察。
- 出口物流、排貨、船務、ERP 的 process patterns。
- 組織接受度與變革阻力案例。
- 育兒、睡眠、家庭溝通等生活原則。
- 文章、書籍、課堂教材中真正能轉成 Sean 自己語言的概念。
- Sean 未來職涯可複用的方法論。
不應直接進 Obsidian：
- 報價 raw data。
- 月用量 raw table。
- 發票明細。
- 客戶訂單狀態。
- 出貨進度。
- 每日 task。
- 即時營運資料。
規則很簡單：
```plain text
會變動的狀態留 Notion / Excel / ERP。
能複用的洞察進 Obsidian。
```
### 4.4 OKF 的權責
OKF 是 agent-readable exchange bundle，不是第二套知識庫。
適合進 `_okf/`：
- 已 mature 的 concept notes。
- 去識別化後的 case notes。
- 可重複執行的 playbooks。
- PMBA / 工作 / AI tooling 的 cross-domain concepts。
- 給 ChatGPT / Claude / Notion agent 使用的 context package。
不應進 `_okf/`：
- private / confidential notes。
- raw transcript。
- customer-sensitive data。
- 還未經 Sean 判斷的 inbox notes。
- Obsidian plugin metadata-heavy notes。

建議：
> **Vault 放 D 槽：****`D:\Knowledge\Sean-KB\`****。**

- Markdown vault I/O 不重，D 槽 SATA SSD 足夠。

### 5.4 OKF-compatible frontmatter
OKF v0.1 要求 concept document 有 YAML frontmatter，且 `type` 是唯一必填；`title`、`description`、`resource`、`tags`、`timestamp` 是建議欄位；unknown fields 應被保留且不該導致 consumer reject 文件。[Source: OKF SPEC](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)
Sean-KB promoted note 應採：
```yaml
---
# OKF-compatible core fields
type: Concept
title: 需求不是任務清單而是決策約束
description: 將需求理解為限制條件、利害關係與取捨，而不是單純任務項目。
resource:
tags:[management, requirements, decision-making]
timestamp: 2026-06-25T10:30:00+08:00

# Sean custom fields
id: pkm-20260625-001
status: seed
domain: work
lang: zh-TW
confidence: medium
sensitivity: internal
source_type: transcript
source_ref:
-"[[SRC PMBA Strategy 2026-06-28]]"
notion_refs:
-"https://www.notion.so/..."
aliases:
- 需求定義
- requirement framing
reviewed:false
reviewed_at:
---
```
### 5.5 OKF concept ID 與 Sean internal ID 的差異
OKF concept ID 是「bundle 內的 path without `.md`」。例如：
```plain text
_okf/concepts/requirement-framing.md
```
OKF concept ID 是：
```plain text
concepts/requirement-framing
```
Sean 的 `id: pkm-20260625-001` 則是內部穩定 ID，用於追蹤原始 note。兩者不可混淆。
建議：
- Obsidian note 可以改名，但 `id` 不變。
- `_okf/` path 盡量穩定，因為 OKF 以 path 作 concept identity。
- 若 `_okf/` path 改名，必須寫入 `_okf/log.md`。

### 6.4 `CLAUDE.md` 與 `.claude/` 的角色

### 8.3 OKF conformance lint
OKF v0.1 的最低 conformance：
1. 每個非 reserved `.md` file 都有 parseable YAML frontmatter。
2. 每個 frontmatter 都有 non-empty `type`。
3. `index.md`、`log.md` 若存在，需符合 reserved filename 用途。
其他如 missing optional field、unknown type、unknown keys、broken links、missing index.md 都應視為 soft guidance，而不是直接拒絕 bundle。[Source: OKF SPEC](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)
建議 lint JSON：
```json
{
  "okf_version": "0.1",
  "bundle_path": "_okf",
  "checked_at": "2026-06-25T10:30:00+08:00",
  "hard_errors": [],
  "soft_warnings": [],
  "sensitivity_warnings": [],
  "recommended_actions": []
}
```

## C. Prompt 與 Skill 文件
### C.1 OKF Exporter Prompt
```plain text
你是 Sean-KB 的 OKF exporter。

請讀取以下 Obsidian note，產生一份 OKF v0.1 compatible concept document。

規則：
1. 輸出標準 Markdown，不使用 Obsidian wikilink。
2. 所有內部連結請轉成 bundle-relative Markdown link，例如 `/concepts/xxx.md`。
3. YAML frontmatter 必須包含：
   - type
   - title
   - description
   - resource
   - tags
   - timestamp
4. 可以保留 Sean custom fields，但不得刪除未知欄位。
5. 若原 note 含敏感資料，請輸出去識別化版本，並標記 sensitivity。
6. 若資料不足以成為 OKF concept，請輸出 reject reason。
7. 每次 export 後更新 `_okf/log.md`。

輸入：
- Obsidian note path
- Obsidian note content
- link mapping table
- sensitivity rule

輸出：
- OKF concept path
- OKF markdown content
- warnings
```
### C.2 OKF Lint Prompt
```plain text
你是 OKF conformance linter。

請檢查 `_okf/` bundle 是否符合 OKF v0.1 的最低規格。

請檢查：
1. 每個非 reserved `.md` 是否有 YAML frontmatter
2. 每個 concept 是否有非空 `type`
3. `index.md` 與 `log.md` 是否符合 reserved filename 用途
4. Markdown links 是否可解析
5. 是否有 Obsidian wikilink 殘留
6. 是否有 missing description / missing timestamp
7. 是否有 path identity 被改動的風險
8. 是否有敏感內容被 export
9. unknown fields 是否被保留

輸出 JSON：
{
  "bundle": "_okf",
  "okf_version": "0.1",
  "hard_errors": [],
  "soft_warnings": [],
  "sensitivity_warnings": [],
  "recommended_actions": []
}
```
### C.3 Inbox Triage Prompt
```plain text
你是 Sean-KB 的 inbox triage agent。

請掃描 `_inbox/`，將每份內容分類為：
1. promote_to_concept
2. promote_to_case
3. promote_to_literature
4. keep_as_source
5. archive
6. delete_candidate

判斷規則：
- 能跨場景複用的想法才 promote。
- 工作 raw status 不 promote。
- 沒有明確 Sean insight 的 highlight 不 promote。
- 涉及 private/confidential 的內容不得進 `_okf/`。

請輸出：
- 建議動作
- 理由
- 需要 Sean 判斷的問題
```

## D. Notion 決策紀錄模板
```markdown
# 決策：Sean-KB 採 OKF-compatible Obsidian LLM Wiki 架構

## 判準

-AI / Agent friendly 優先。
-保留 Obsidian local-first 與 filesystem-native 優勢。
-Notion 繼續作為任務、專案與決策控制檯。
-不讓 Notion / Obsidian 做內容雙向同步。
-讓知識庫可被不同 agent 讀取、lint、export。
-不增加 Sean 的手工維護負擔。

## 決策

採用 OKF v0.1 作為 Sean-KB 的 agent exchange layer。Obsidian 原生 vault 仍是長期知識 SoT；新增 `_okf/` 作為由 AI / script 產生的 OKF-compatible bundle，不手工維護，不作為第二套知識庫。

## 影響

-Frontmatter schema 加入 OKF core fields：
  -type
  -title
  -description
  -resource
  -tags
  -timestamp
-保留 Sean custom fields：
  -id
  -domain
  -status
  -lang
  -confidence
  -sensitivity
  -source_ref
  -notion_refs
-新增 OKF export / lint prompt。
-Notion ↔ Obsidian 橋接仍採雙向異步引用，不做內容同步。
-`_okf/` 作為 agent-readable / exchange / lint layer。

## 行動

1.建立 `D:\Knowledge\Sean-KB\`。
2.更新 Obsidian note template。
3.建立 `CLAUDE.md`。
4.建立 `_okf/` skeleton。
5.建立 `_system/prompts/okf-exporter.md`。
6.建立 `_system/prompts/okf-lint.md`。
7.第 1 週只讓 PMBA 首堂課產出的卡片符合 OKF-compatible schema。
8.30 天後再評估是否擴大 export 範圍。

## 非目標

-不把 OKF 當成第二套筆記系統。
-不把全部 Obsidian vault 硬改成 OKF。
-不把 Notion 內容完整同步到 Obsidian。
-不導入本地 LLM 作為主路徑。
-不追求 graph view 漂亮。
```

# 最終結論
v1.1 的方向沒有錯，但它漏掉了「完整架構」最重要的部分：工具層、agent 入口、治理層、Notion-side workflow、PMBA 與工作 domain pipeline。v1.2 的正式結論是：
```plain text
Sean-KB = Obsidian-native long-term knowledge base
        + Notion action / project / decision console
        + OKF generated exchange bundle
        + Claude Code filesystem maintainer
        + Notion MCP / ntn / Workers as Notion-side bridge
        + Sean as curator and decision owner
```
最小可行行動不是架一堆東西，而是：
1. 建立 `D:\Knowledge\Sean-KB\`。
2. 建立 folder skeleton。
3. 建立 `CLAUDE.md`。
4. 建立 OKF-compatible note template。
5. 第 1 週只處理 PMBA 首堂課與 1 個工作案例。
6. 第 1 個月只累積 20–30 張高品質 notes。
7. 第 2 個月開始導入 lint / OKF export。
8. 第 3 個月讓 Notion ↔︎ Obsidian ↔︎ OKF 穩定互引。
這樣 Sean 不會多一個要手工維護的筆記系統，而是會得到一套可以被 AI 長期維護、可審查、可匯出、可回到行動的個人知識作業系統。
